## 1. 출력 스키마 (C1)

```json
{
  "verdict": "smishing | suspicious | normal",
  "risk_factors": ["impersonation_delivery", "url_shortener"],
  "explanation": "택배사를 사칭하며 공식 도메인이 아닌 단축 URL로 클릭을 유도하는 전형적인 스미싱입니다. 실제 택배사는 문자로 앱 설치나 개인정보 입력을 요구하지 않습니다."
}
```

- `verdict`: 3클래스 판정
  - `smishing` — 기만적 의도가 명확 (사칭 + 링크/행동 유도)
  - `suspicious` — 위험 신호가 있으나 단정하기 어려움 (예: 출처 불명 링크만 존재)
  - `normal` — 정상 (합법 광고 포함, §3 참조)
- `risk_factors`: 아래 태그 목록에서 해당하는 것만 배열로. `normal`이면 빈 배열 `[]`
- `explanation`: 2~3문장. ① 문자의 어떤 부분이 왜 위험/안전한지 + ② 사용자가 배울 수 있는 일반 원칙 1개
- 신뢰도(confidence)는 **생성하지 않음** — 모델이 지어내는 숫자는 무의미. 필요 시 verdict 토큰의 logit에서 산출 (데모 단계에서 결정)

### risk_factors 태그 목록 (15개, 5개 축)

| 축 | 태그 | 의미 |
|----|------|------|
| URL/링크 | `url_shortener` | 단축 URL 사용 (bit.ly, han.gl, me2.kr 등) |
| | `suspicious_domain` | 공식 도메인이 아니거나 오타를 흉내낸 도메인 |
| 사칭 | `impersonation_delivery` | 택배·물류 사칭 (주소지 오류, 배송 조회) |
| | `impersonation_gov` | 정부·수사기관·건강보험·법원 사칭 |
| | `impersonation_financial` | 은행·카드사·증권사 사칭 |
| | `impersonation_acquaintance` | 지인·가족 사칭 ("엄마 폰 고장났어", 부고, 청첩장) |
| 심리 조작 | `urgency_pressure` | 시간 압박 ("오늘까지", "즉시 확인") |
| | `fear_appeal` | 불안 조성 (벌금, 계정 정지, 범죄 연루, 미납) |
| | `financial_lure` | 금전 유인 (환급금, 정부지원금, 고수익 투자) |
| 요구 행동 | `app_install_request` | 앱(APK) 설치·업데이트 유도 |
| | `personal_info_request` | 개인정보·인증번호·계좌정보 요구 |
| | `callback_induce` | 낯선 번호로 회신 전화 유도 |
| 형식 | `abnormal_format` | 자모 분리, 특수문자 삽입 등 필터 우회 표기 |
| | `unverifiable_sender` | 국제발신, 발신 주체 확인 불가 |
| | `link_only_message` | 맥락 설명 없이 링크만 포함 |

- 태그는 증강 시 **유형 축**(사칭 대상 × 수법 × 말투)으로도 사용 → 다양성 확보와 중복 제거의 기준
- 실데이터 EDA 후 태그 추가/병합 가능 (버전 기록)

## 2. 학습 프롬프트 템플릿 (C2)

chat 형식 (system / user / assistant). SFTTrainer의 chat template 적용을 전제로 함.

**system**

```
당신은 한국어 문자 메시지의 스미싱(문자 사기) 여부를 분석하는 보안 도우미입니다.
문자를 분석하여 반드시 아래 JSON 형식으로만 답하세요.
{"verdict": "smishing|suspicious|normal", "risk_factors": [...], "explanation": "..."}
- verdict: smishing(사기), suspicious(의심), normal(정상) 중 하나
- risk_factors: 발견된 위험 요인 태그 배열 (정상이면 빈 배열)
- explanation: 판단 근거를 2~3문장으로, 사용자가 앞으로 유사한 문자를 스스로 판별할 수 있도록 설명
```

**user**

```
다음 문자 메시지를 분석해 주세요.

문자: {text}
```

**assistant**

```
{"verdict": "...", "risk_factors": [...], "explanation": "..."}
```

- 학습·평가·데모 모두 동일 템플릿 사용 (train/serve 불일치 방지)
- zero-shot 베이스라인 측정 시에도 같은 system 프롬프트 사용 — 파인튜닝 효과만 분리하기 위함

## 3. 정상(normal) 라벨 기준 (B3)

**원칙: verdict는 "기만성" 기준이지 "광고 여부" 기준이 아니다.**
합법 광고(수신거부 표기, 실제 사업자, 공식 도메인)는 `normal`이다. 광고를 스미싱으로 배우면 오탐(false positive)이 폭증하고, "차단 사유를 설명한다"는 제품 약속과도 어긋난다. 반대로 발신 주체를 사칭하거나, 링크·전화·앱 설치로 유도하면서 그 목적을 속이는 문자는 광고 형식이어도 `smishing`이다. 그 사이 — 기만 신호가 일부 있으나 사칭·사기 의도를 단정할 수 없는 경우(출처 불명 링크, 확인 불가 발신자)가 `suspicious`다.

경계 사례 처리:

| 사례 | 라벨 | 근거 |
|------|------|------|
| 수신거부 표기 있는 대출 광고 | `normal` | 합법 광고. 단 explanation에 "금리 조건 확인 필요" 등 주의점 언급 가능 |
| 공식 도메인의 카드 승인 알림 | `normal` | 발신 주체와 내용 일치 |
| 링크만 있고 설명 없는 문자 | `suspicious` | `link_only_message` — 사칭 여부 확인 불가 |
| 실제 기관명 + 비공식 도메인 | `smishing` | 사칭 확정 신호 |

정상 데이터 소스(우선순위): ① Kor-Smishing ham(확보 시) ② 본인 수신 문자 익명화(광고 포함 — 경계 사례 확보 목적) ③ 공개 한국어 대화 코퍼스(문자체 문장만 선별). 개인 수신 문자는 발신번호·이름·주소 등 개인정보 제거 후 저장.
